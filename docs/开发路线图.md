## 开发前参考要求

后续实施本路线图中的任一机制时，必须先按“参考学习 → 本仓库现状核对 → 独立实现设计”的顺序推进：

1. 只读查看 `docs/external-references/Reasonix` 与/或 `docs/external-references/claude code` 中对应机制的设计、边界和流程。
2. 回到本仓库真实源码，确认 `src/main/`、`src/preload/`、`src/renderer/`、`src/shared/`、`tests/` 与相关文档中的现有类型、函数、IPC、事件和持久化结构。
3. 输出本次机制的“改什么 / 为什么改 / 影响面多大”，再进入代码或文档修改。
4. 只借鉴设计，不复制、链接、导入或构建外部参考源码；所有运行时代码必须在本仓库 `src/` 下独立实现。

## 一、学习笔记

### Reasonix

**一句话定位**

Go 实现的配置/插件驱动 coding agent，面向 CLI、Desktop、ACP、Bot 多前端复用同一运行核心。

**核心模块划分**

- 前端无关 Controller、`internal/agent` 运行循环、`internal/control`
- `internal/provider` 模型注册表；`internal/tool` 工具注册表
- `internal/plugin` MCP client；`internal/permission` 权限规则
- `internal/checkpoint` 快照回滚；`internal/skill` 技能系统
- `internal/command` slash command；`internal/event` typed event stream

**值得借鉴的机制设计（高价值）**

- 前端无关 Controller；Provider/Tool registry
- MCP 插件 host；per-call permission rule
- checkpoint + rewind；planner/executor 双模型分离会话
- skill/subagent；自定义 slash command
- typed event sink；工具进度事件；上下文压缩归档；工具调用历史修复

**值得借鉴的逻辑构成**

- 运行链路：boot → Controller → Agent/Coordinator → Provider/ToolRegistry → event.Sink
- 工具执行前后统一走 gate/hooks/preview/checkpoint
- 插件连接用并发上限、单插件超时和失败隔离
- planner 只拿只读工具，executor 执行写入
- checkpoint 每 turn 第一次触碰文件存 pre-edit snapshot

**不建议借鉴**

> Go/Wails/CLI/Bubble Tea 具体实现、DeepSeek 专属缓存策略、业务 UI、Bot 平台适配、macOS Seatbelt 细节不应直接迁移。
> 

### claude code

**一句话定位**

个人 Claude Code 工作流样本，不是完整应用源码。

**核心模块划分**

- `.claude/agents` greeting agent；`.claude/commands` interview command
- `.claude/skills/teach-me` 教学技能；`teach-me/records` 学习状态记录
- `.vscode` 调试配置；`bun.lock` 依赖快照

**值得借鉴的机制设计（高价值）**

- 技能文档结构；问答式需求访谈
- 学习/状态记录以 Markdown 持久化；按 topic slug 管理 session/profile
- 用强规则约束 agent 行为

**值得借鉴的逻辑构成**

- teach-me 流程：输入解析 → 诊断水平 → 构建概念列表 → 问答循环 → 记录误区/掌握状态
- interview 用强制结构化问题驱动需求澄清

**不建议借鉴**

> `bun.lock` 只是依赖快照，不能代表完整架构；`.vscode` 调试配置和个人学习记录不适合作为项目机制迁移来源。
>
